Go: Writing unit tests in Go

September 15, 2019

Go Testing

Unlike other languages, Go provides the testing framework in its standard library making testing in Go much simpler and indeed a must in anything going to production.

Writing tests are easy as Go coding in general. Lets see a simple test that checks a greeting function produces the correct output.

TDD Process to Get a Passing Test

Writing Failing Test

package main

import "testing"

func TestSimpleGreeting(t *testing.T){

    got := greetMe("Keir")
    want := "Welcome, Keir"

    if got != want {
        t.Errorf("wanted %s, but got %s", want, got)
    }
}

main_test.go

The testing package is imported from the standard library, providing a plethora of methods we can use.

Tests in Go must start with the word Test with the first letter capitalised. Otherwise the test will not run when called. For example TestSimpleGreeting().

The test needs to take in a pointer to the testing method T, which is commonly assigned to the variable t.

Save the file with the suffix _test.go for example: main_test.go - this is so the Go testing tool knows that this Go file contains tests it should be compiling and running.

Running the Go testing tool

$ go test *.go
# command-line-arguments [command-line-arguments.test]
./main_test.go:7:9: undefined: greetMe
FAIL	command-line-arguments [build failed]

Here we can see the test failing as the greetMe func doesn’t exist.

Now we have a failing test for something we want to achieve, we now have to write the minimal code to get the test to pass.

package main

func greetMe() (greeting string) {
    return
}

main.go

Running the tests again gives..

$ go test -v *.go
# command-line-arguments [command-line-arguments.test]
./main_test.go:7:16: too many arguments in call to greetMe
        have (string)
        want ()
FAIL    command-line-arguments [build failed]

The test is showing that we are passing in a string to the func call, but it wasn’t expecting one.

Lets fix this

package main

func greetMe(name string) (greeting string) {
	return
}

main.go

Run the test again..

$ go test -v *.go
=== RUN   TestSimpleGreeting
--- FAIL: TestSimpleGreeting (0.00s)
    main_test.go:11: wanted Welcome, Keir, but got 
FAIL
FAIL    command-line-arguments  0.614s

The tests show we are expecting “Welcome, Keir” to be returned, but got nothing back.

Lets fix this…

package main

func greetMe(name string) string {
	return "Welcome, " + name
}

main.go

Running the test one more time…

$ go test -v *.go
=== RUN   TestSimpleGreeting
--- PASS: TestSimpleGreeting (0.00s)
PASS
ok      command-line-arguments  (cached)

Hurrah! Our simple test is now passing.

Check our Code Coverage

Running the Go test tool with the -cover flag will show you how many of the statements are being covered by a test. For example:

$ go test -cover *.go
ok      command-line-arguments  0.522s  coverage: 100.0% of statements

Unsurprisingly our very simple example is showing 100% code coverage; but you could imagine this creeping lower as you Go project go more complex.

This tool will allow you to keep an eye on your test coverage and even see which lines of code aren’t being called.
